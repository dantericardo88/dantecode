DanteCode: Production-Ready Product Requirements Document (PRD)D1 Executive Summary & VisionThe landscape of autonomous artificial intelligence coding agents has reached a critical inflection point, currently defined by a stark bifurcation between highly polished, closed-ecosystem, vendor-locked solutions and fragmented, open-source experimental frameworks. Enterprise engineering teams increasingly demand the seamless, stateful orchestration provided by proprietary tools, yet they remain constrained by forced model dependencies, unpredictable pricing, and the inherent lack of verifiable, deterministic quality gates. DanteCode is engineered to bridge this divide by unifying the industry's most advanced open-source repositories into a single, cohesive, production-ready product. This document serves as the exhaustive Product Requirements Document (PRD) for DanteCode, defining its architecture, core features, technical specifications, and comprehensive implementation roadmap.DanteCode is envisioned as a fiercely model-agnostic, open-source AI coding agent that delivers the fluid user experience and stateful orchestration characteristic of premium closed-source competitors, while completely eliminating their critical weaknesses. By defaulting to the ultra-fast Grok API, specifically the grok-4.20-beta-latest-non-reasoning and grok-4-1-fast-reasoning models, DanteCode ensures maximum execution speed, a massive 2,000,000 token context window, and unparalleled cost efficiency. The architecture simultaneously maintains robust compatibility with all major Large Language Models (LLMs), ensuring engineering teams are never locked into a single provider infrastructure.The core architectural paradigm is an accelerated, precise fusion methodology—a clean merge of five premier open-source initiatives. The base skeleton, Command Line Interface (CLI), Language Server Protocol (LSP) integrations, and multi-model routing capabilities are harvested directly from OpenCode. The Visual Studio Code (VS Code) extension richness, including provider-agnostic graphical user interfaces, inline diff editing, and persistent chat sidebars, is sourced from Continue. Git-native operations, automatic commit message generation, repository mapping via Tree-sitter, and precision multi-file diff reviews are integrated from Aider. The secure, sandboxed execution runtime, which facilitates stateful agent loops and secure tool-calling environments via isolated Docker containers, is provided by OpenHands. Finally, DanteForge serves as the disciplined, deterministic brain of the system, injecting the Project Documentation/State Evaluation (PDSE) scoring engine, the Autoforge Intelligent Autonomous Loop (IAL), rigid verification gates, and the uncompromising Anti-Stub Doctrine.Furthermore, DanteCode leverages highly optimized internal system prompt mechanics reverse-engineered from industry-leading tools. This entails implementing precise todo_write and todo_read task tracking protocols, orchestrating parallel sub-agent deployments known as "Agent Teams," and enforcing stringent context management protocols that prevent hallucination and context degradation over long-running sessions. A mandatory, central feature of this product is the automated dantecode skills import --from-claude command. This feature seamlessly ingests, parses, adapts, and mathematically verifies existing Claude Code skills, ensuring zero migration friction for enterprise users transitioning to the DanteCode ecosystem. The resulting system is not merely a wrapper around an LLM, but a deterministic, mathematically gated software engineering pipeline that guarantees code quality before any artifact is permitted to merge into the main branch.D2 Architecture & Tech StackThe DanteCode architecture is constructed as a highly modular, decoupled system designed for robust local execution and secure, high-throughput integration with cloud-based LLM APIs. The technology stack relies heavily on TypeScript and Node.js for the core orchestrator, Python for specific sandboxed evaluation environments, and Docker for absolute runtime isolation. The system is divided into five specialized layers, each responsible for a distinct phase of the autonomous development lifecycle.The Interface Layer is built upon the foundations of OpenCode and Continue. The Terminal User Interface (TUI) provides an interactive, terminal-native experience with full ANSI color support, allowing developers to execute workflows directly from their command line. The Integrated Development Environment (IDE) extension operates as the primary Graphical User Interface (GUI), featuring a persistent sidebar, inline diff highlighting, and Fill-In-The-Middle (FIM) autocomplete models designed to operate with minimal latency. A Language Server Protocol (LSP) implementation manages workspace synchronization and file diagnostics, ensuring the agent remains perpetually aware of the current project state.The Orchestration and Brain Layer is entirely governed by DanteForge. It replaces traditional, unpredictable stochastic agent loops with a rigid, deterministic state machine (src/core/autoforge.ts). This state machine enforces a strict workflow progression: Review, Constitution, Specify, Clarify, Plan, Tasks, Forge, Verify, and Synthesize. The Intelligent Autonomous Loop (IAL v2) handles autonomous progression through these stages, but only if the PDSE Score Engine permits it. The PDSE engine continuously evaluates the quality, completeness, and clarity of project artifacts, generating quantitative scores that gate phase progression. If an artifact scores below the mandated threshold, the loop halts, and the LLM is prompted to self-correct the specific bottleneck.Architectural LayerCore TechnologyPrimary ResponsibilitySourced RepositoryInterface LayerTypeScript, React, LSPTUI/GUI presentation, inline editing, user interactionOpenCode, ContinueOrchestration LayerTypeScript, State MachineDeterministic workflow gating, PDSE scoring, IAL v2DanteForgeContext LayerPython, Tree-sitter, GitAST repository mapping, diff generation, auto-commitsAiderExecution LayerDocker, PythonSandboxed terminal execution, secure tool callingOpenHandsAPI Routing LayerREST, Server-Sent EventsLLM payload formatting, streaming, multi-model routingOpenCodeThe Context and Version Control Layer is powered by the precise algorithms developed within Aider. Autoforge Wiring & Intelligent Autonomous Loop LogicThe dantecode autoforge command serves as the primary engine operating the Intelligent Autonomous Loop (IAL). The architectural wiring between the CLI parsing, the PDSE scoring engine, and the execution runtime is highly deterministic.Initialization begins in src/core/autoforge.ts, where the CLI parses options such as --dry-run, --max-waves, and --auto. The system loads the STATE.yaml file into memory and instantiates the AutoforgeContext. Before any action is dispatched to the LLM, the loop triggers a comprehensive PDSE scan (src/core/pdse.ts). The PDSE engine parses the Markdown artifacts on disk. For instance, it uses regular expressions to verify that TASKS.md contains the mandatory ### Phase headers.If any critical artifact scores below the acceptable threshold of 50, or if the DecisionStatus evaluates to pause, the loop halts immediately. It externalizes the bottlenecks to the user or passes them to the Lead Agent for self-correction. If the DecisionStatus is advance, the loop determines the next logical WorkflowStage. It constructs a prompt combining the CONSTITUTION.md, the current artifact contents, and the tool JSON schemas. After LLM execution, the loop dispatches a command to the OpenHands Docker container to run the designated test suite. The results of this sandboxed execution are fed back into the PDSE engine to update the lastVerified timestamp.Grok API Integration DetailsTo systematically eliminate dependency on Anthropic's infrastructure, DanteCode defaults to xAI's Grok API, specifically utilizing grok-4-1-fast-reasoning for complex logic and grok-4.20-beta-latest-non-reasoning for rapid parsing tasks. Before transitioning from the Forge execution stage to the Synthesize review stage, DanteCode is mandated to execute the project's native test suite (e.g., npm run verify, cargo test, pytest) entirely within the OpenHands sandbox. To ensure foundational stability, npm run check:cli-smoke runs on every pull request, verifying that core routing, state generation, and Grok API connectivity remain intact under load. While acceptable in conversational interfaces, this behavior severely degrades and often catastrophically halts autonomous execution pipelines. To combat this, DanteCode enforces a draconian, bypass-proof Anti-Stub Doctrine.Enforcement Mechanics and AST ParsingThe Anti-Stub Doctrine is not a mere prompt instruction; it is a hard-coded mechanical gate enforced during the Verification phase, prior to invoking any compilers or test suites. DanteCode utilizes Tree-sitter to parse the Abstract Syntax Tree (AST) of the generated code, combining structural understanding with high-speed regex scanning.The AST scanner explicitly hunts the generated diffs for forbidden patterns:Keywords: TODO, FIXME, HACK.Language-specific omissions: pass or ... in Python.Explicit non-implementations: throw new NotImplementedError() in JavaScript/TypeScript, or unimplemented!() in Rust.Semantic comments indicating deferred work: // add logic here, /* to be implemented */.If any of these patterns are detected within the code generated during the current execution wave, a severe PDSE penalty is applied. The Clarity Score of the current phase is instantly hard-coded to 0. A Clarity Score of 0 mathematically forces the PDSE DecisionStatus to evaluate to pause. Because the state machine is deterministic, the Git worktree mathematically cannot be merged into the main branch while a pause state is active.Upon entering this paused state, the Intelligent Autonomous Loop intercepts the decision. It extracts the exact line number and file path of the offending stub. The loop then automatically generates a highly penalized self-correction prompt and forces the LLM into a remediation cycle: "CRITICAL ERROR: Code contains forbidden stubs/placeholders at line X. The Anti-Stub Doctrine requires full, complete implementation. Do not use TODOs. Rewrite the function with actual logic." By inextricably tying the total absence of stubs directly to the mathematical gating of the PDSE engine, DanteCode guarantees that only complete, fully functioning code ever reaches the repository's main branch.
